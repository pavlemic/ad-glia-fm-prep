# NUMPY POLICY — RIDE COLAB'S NATIVE numpy-2 BASE. DO NOT force numpy 1.x here.
#
# History (2026-07-04): the first draft of this file pinned numpy==1.26.4 to
# "avoid numpy-2 surprises." That pin was a pre-commitment from May 2026, made
# before Colab's base image moved to a numpy-2-native scientific stack. On the
# first real run it caused TWO cascading failures, both of them artifacts of
# fighting the platform, not of Geneformer:
#   1. scipy 1.16.3 (Colab's default) + forced numpy 1.26.4 = scipy/scipy#22877
#      ("sph_legendre_p ... All ufuncs must have type numpy.ufunc" on import) —
#      a bug specific to the scipy>=1.15 + numpy==1.26.4 combination.
#   2. After Geneformer's `pip install .` bumped numpy to 2.0.2, a helper in
#      scanpy's import chain was left at a pre-numpy-2 version and raised
#      `AttributeError: module 'numpy' has no attribute 'bool'`.
# Decisive fact: Geneformer's own installer CHOSE numpy 2.0.2 (its setup.py does
# not cap numpy<2), and its heavy deps (transformers 5.x, torch 2.11, datasets
# 4.0) are all numpy-2 native. numpy 1.26 is EOL per SPEC-0. So nothing in this
# pipeline actually needs numpy 1.x — we let Colab's coherent numpy-2 base stand
# (numpy 2.0.2 / scipy 1.16.x / scanpy 1.12.x / anndata 0.12.x) and install
# Geneformer on top of it. No numpy pin, no scipy pin, no constraints file.
#
# REPRODUCIBILITY: this file therefore under-specifies on purpose until the
# first clean run. colab_09 §2a does `pip freeze` and records the exact resolved
# numpy-2 stack + Geneformer commit; copy those pins back here and lock the file
# AFTER a green run. (Same freeze-then-lock plan as before — only the numpy
# target changed from 1.x to 2.x.)
# ---------------------------------------------------------------------------

# TRANSFORMERS v5 PIN — 2026-07-04. Geneformer's setup.py leaves `transformers`
# completely unconstrained, so on the first real run pip resolved v5, which
# broke Geneformer's own geneformer/collator_for_classification.py: it imports
# `SpecialTokensMixin` directly from the transformers top level, but v5 merged
# that class into PreTrainedTokenizerBase and dropped the standalone export
# (see MIGRATION_GUIDE_V5.md). This is an upstream Geneformer/transformers
# version lag, not a numpy or platform issue — the numpy-2 diagnosis above
# still holds. Pin to the last pre-v5 release; installed here BEFORE
# Geneformer's `pip install .`, so pip's resolver leaves it alone (Geneformer's
# own unconstrained requirement is satisfied by any installed version).
# If peft/accelerate/datasets (also unconstrained in Geneformer's setup.py)
# turn out to require transformers>=5, pip will raise a resolver conflict here
# instead of the silent runtime break above — that's the useful failure mode.
transformers==4.57.0
# ---------------------------------------------------------------------------

# ---------------------------------------------------------------------------
# DATASETS VIDEOREADER PIN — 2026-07-04. §5b (EmbExtractor.extract_embs, which
# uses a `datasets`-backed torch Dataset under the hood) crashed with
# `ImportError: cannot import name 'VideoReader' from 'torchvision.io'`.
# Root cause: torchvision >=0.24 removed its video-decoding API (VideoReader
# et al.) entirely in favor of TorchCodec, but the `datasets` version that
# Geneformer's unconstrained install resolved still does an UNGUARDED
# `from torchvision.io import VideoReader` in
# formatting/torch_formatter.py._tensorize whenever torchvision happens to be
# importable — we never touch video data, torchvision is only in sys.modules
# as a side effect of another import in the chain. Confirmed the tagged
# datasets==5.0.0 source wraps that import in try/except ImportError (fixed
# upstream); pin to it. Installed here before Geneformer's `pip install .`,
# same "unconstrained dep, installed first, resolver leaves it alone" logic
# as the transformers pin above.
datasets==5.0.0
# ---------------------------------------------------------------------------

# ---------------------------------------------------------------------------
# TORCHAO REMOVAL — 2026-07-10 (colab_11 §5a). Colab preinstalls torchao 0.10.0,
# which this pipeline never uses (no quantization, plain LoRA via peft). peft's
# get_peft_model dispatches every target Linear through dispatch_torchao(),
# which calls is_torchao_available() — confirmed against peft 0.19.1 source
# (peft/import_utils.py): if torchao is merely *importable*, that function
# hard-raises ImportError below its 0.16.0 floor instead of returning False.
# This is not expressible as a requirements.txt pin (there is nothing to
# require — we want it ABSENT), so the notebook runs
# `!pip uninstall -y torchao -q` after Geneformer's `pip install .` (§1a).
# Uninstalling rather than upgrading to >=0.16.0 avoids adding a new floated
# dependency with its own torch-version coupling for a package never used.
