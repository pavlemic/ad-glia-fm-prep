# REVERSAL — 2026-07-04. The previous version of this file pinned the full old
# scGPT stack (scgpt 0.2.5 + scvi-tools==0.20.3 + scanpy==1.9.8 + numpy 1.26.4,
# etc.) and declared this env needed a Python-3.10 runtime on Colab, because
# scGPT's PACKAGING declares scvi-tools<1.0 (which caps Python <3.11). That plan
# was abandoned: Colab's default runtime moved to Python 3.12 and every attempt
# to obtain a real 3.10 kernel on Colab (condacolab, colab-python-version) was
# unreliable.
#
# Decisive fact (from reading the scGPT source at the pinned commit, not its
# packaging): scGPT's DECLARED dependencies are far broader than what its
# load+embed code actually IMPORTS.
#   - scvi-tools : declared in pyproject, imported NOWHERE in the package. Phantom.
#   - torchtext  : guarded by scgpt/tokenizer/vocab_compat.py — falls back to a
#                  pure-Python gene Vocab when torchtext is absent. Loading a
#                  pretrained vocab.json never touches it.
#   - flash-attn : guarded by scgpt/model/flash_attn_compat.py — falls back to
#                  standard PyTorch attention (identical maths, slower).
#   - scib       : imported inside a function (eval_scib_metrics), off the
#                  embedding path entirely.
# So scGPT is installed with `--no-deps` from a pinned commit, and only the
# packages its import chain really needs are added on top of Colab's native
# base. The old scvi-tools<1.0 -> Python-3.10 chain simply does not apply.
#
# NOTE: this native-Python approach is validated by colab_10 §1–§5 running
# green (import scgpt + embed). Until that run is confirmed, treat this file as
# the intended env, not a proven one. (Same freeze-then-lock discipline as the
# Geneformer env: it under-specifies on purpose until the first clean run.)
#
# UPDATE 2026-07-07 — §1–§2 gate CONFIRMED GREEN (`import scgpt`/`scanpy`/
# `anndata` all resolve natively). One extra fix was needed beyond the above:
# see the typing_extensions pin below. §3–§7 (load, vocab audit, embed) not yet

# Real runtime deps not preinstalled on Colab's base (installable via
# `pip install -r`):
scanpy                          # not in Colab base; float on Colab's coherent stack
anndata                         # not in Colab base; float
psutil                          # RAM logging helper the notebooks use directly

# datasets==5.0.0 — HARD PIN, but for an unusual reason: `import scgpt` eagerly
# imports scgpt.scbank, which does `from datasets import Dataset, load_dataset`
# at module load, so the NAME must resolve for the import to succeed. scGPT's
# packaging asks for datasets>=2.20,<3, but (a) the zero-shot embedding path
# never actually CALLS scbank/datasets, so only the import needs to work, and
# (b) 2.x fights Colab's py3.12/numpy-2 base. 5.0.0 is the version already
# validated in the Geneformer env (colab_09) and it exposes both names — pinned
# here for that import + cross-FM-env consistency. If `import scgpt` ever fails
# specifically on datasets, that is the first knob to turn (colab_10 §1b names
# the fallback).
datasets==5.0.0

# typing_extensions>=4.16.0 — HARD PIN, found 2026-07-07. Colab's base shipped
# anndata 0.13.0 alongside typing_extensions 4.15.0, but anndata 0.13.0's
# `anndata/acc/__init__.py` does `from typing_extensions import sentinel`,
# which 4.15.0 doesn't have (only the older `Sentinel` class) -> ImportError
# kills anndata -> kills scanpy (imports anndata.logging) -> kills scgpt
# (imports AnnData). Nothing to do with scGPT's own stack; a Colab-native-base
# version-skew, same failure class as colab_09's transformers/torchvision
# breaks. Fix: upgrade to 4.16.0+ (first version with `sentinel`). Listing it
# explicitly here (rather than leaving it an unpinned transitive dep) is what
# makes pip actually upgrade it instead of treating Colab's stale 4.15.0 as
# already satisfying anndata's requirement.
typing_extensions>=4.16.0
